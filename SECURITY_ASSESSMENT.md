# セキュリティ評価（ローカル運用前提）

## 1. 評価対象
- アプリ: `the-test`（Electron + ローカル SQLite）
- 想定運用: インターネット非接続、単一PCでのローカル利用
- 保存データ: テストケース/シナリオ/実行結果、証跡ファイル（画像・添付）

## 2. 評価前提（依頼条件）
- 評価対象外:
  - 画面の覗き見などソーシャルエンジニアリング
  - OS/端末のマルウェア感染
- 評価観点:
  - 上記を除いた場合に、業務情報が外部へ漏えいしにくいか

## 3. 実装確認結果（コードベース）
- 外部通信:
  - 通常機能で `fetch`/HTTP送信は実装されていない
  - データはローカルファイル入出力（SQLite/エクスポート/添付）中心
- Electron分離設定:
  - `contextIsolation: true`
  - `nodeIntegration: false`
  - レンダラからは `preload` で公開したIPC APIのみ利用
- データ保管:
  - DBは `.sqlite` ファイル、証跡はプロジェクト配下ファイルとして保存
  - バックアップは指定フォルダにコピー

## 4. リスク評価
- 漏えい経路（ネットワーク）:
  - アプリ単体での自動送信機能は見当たらず、低リスク
- 端末内での情報露出:
  - データが平文（SQLite/ファイル）で保存されるため、端末アクセス権のある利用者には閲覧可能
- 入力経由の安全性:
  - IPCパラメータの厳格バリデーションは限定的
  - `shell.openPath` はローカルファイルオープン用途で外部URL直接起動ではない
- レンダラ防御:
  - `index.html` にCSP未設定（現状はローカル実行中心で直ちに重大ではないが、防御層としては弱い）

## 5. 判定
- 判定: **条件付きで業務利用可能**
- 理由:
  - オフライン前提かつ現行実装では外部送信経路がほぼないため、「外部漏えい」観点では実用レベル
  - ただし、保存データが平文であるため、端末利用権限を持つ内部者への耐性は限定的

## 6. 業務利用レベルにするための推奨対策（優先順）
1. 保存データ暗号化
   - DB/証跡ファイルの暗号化（少なくともディスク暗号化 + アプリ起動時パスフレーズ保護）
2. Electronハードニング
   - `BrowserWindow` に `sandbox: true` を検討
   - `index.html` へCSP導入（`default-src 'self'` など）
3. IPC入力バリデーション
   - `main` 側で payload schema バリデーションを追加し、想定外入力を拒否
4. 監査性
   - 重要操作（削除、エクスポート、バックアップ）の監査ログ出力
5. 運用統制
   - プロジェクト保存先をアクセス制御された業務領域に限定
   - バックアップ媒体の保管ルール（暗号化・持ち出し管理）を明文化

## 7. 結論
- 「ローカルのみ・外部送信なし」という前提では、現状でも外部漏えいリスクは低い。
- 会社業務で継続利用するなら、**暗号化/CSP/IPCバリデーション**を入れることで、セキュリティ説明責任を満たしやすくなる。
